## 核心更新说明
本次更新已完成“用户行为记录”与“衣服全局统计”系统的数据库设计及接口开发。数据已与现有的 `clothing_features` 表通过 `filename` 字段实现对齐。



## 1. 数据库设计 (PostgreSQL)
共新增两张表，已在 Docker 环境中部署完成：
> 这里我已经更新init.sql了

### A. 用户行为明细表 (`user_item_behavior`)
*   **用途**：记录每个用户产生的每一条交互数据（推荐系统原始权重数据）。
*   **核心字段**：`user_phone`, `filename` (衣服名), `stay_duration` (秒), `is_like`, `is_collect`, `is_share`。

### B. 衣服全局统计表 (`item_engagement_stats`)
*   **用途**：汇总全平台对单件衣服的交互总量（前端详情页展示用）。
*   **核心字段**：`filename` (主键), `total_likes`, `total_collects`, `total_shares`, `total_views`, `total_stay_time`。

### C. 用户基础信息统计表 (`users`)
*   **用途**：替代user.csv。
*   **核心字段**：`phone`(作为唯一标识)、`password`、`user_id`、`created_at`、`gender`、`age`、`height`、`weight`、`body_type`、`style_preferences`、`analysis_records`
> 这里users表的字段设计对齐user.csv

## 2. 接口文档
新接口在swagger文档的 用户交互统计接口 和 服装信息统计接口里面了

### ① 行为数据上报 (交互/离开页面时调用 -> ai给的逻辑是说计算进入和退出时间算出页面停留时间)
> 已经支持取消点赞收藏的功能，不是采用事件触发逻辑是状态同步逻辑
*   **接口地址**: `POST /api/behavior/record`
*   **逻辑说明**: 
    *   状态同步逻辑：本接口不只是“记录”，而是“同步”。
    *   无需独立取消接口：若用户取消点赞，前端只需发送 is_like: false，后端会自动计算该用户上一次的状态。若上一次为 true 这一次为 false，则统计表中的 total_likes 会自动减 1。
    *   调用时机：建议在用户点击交互按钮，或关闭详情页弹窗时统一上报。
*   **Payload 示例**:
    ```json
    {
    "user_phone": "13800000000",
    "filename": "img_0001.jpg",
    "stay_duration": 15,
    "is_like": true,  // 传 true 为点赞，传 false 为未点赞或取消点赞
    "is_collect": false,
    "is_share": false
    }
    ```

### ② 获取衣服汇总数据 (进入详情页时调用)
*   **接口地址**: `GET /api/item/stats/{filename}`
*   **返回示例**:
    ```json
    {
      "status": "success",
      "data": {
        "total_likes": 128,
        "total_collects": 45,
        "total_shares": 12,
        "total_views": 1024,
        "total_stay_time": 5600
      }
    }
    ```

## 前后端联调建议 (AI给的)
1.  **字段对齐**：所有接口的衣服标识符统一使用 **`filename`** (Text 类型，如 `img_0001.jpg`)，确保与您前端现有的图片加载逻辑一致。
2.  **时长计算**：建议在用户进入详情页时记录 `startTime`，离开/关闭弹窗时计算 `duration = (now - startTime) / 1000` 并通过 POST 接口发送。
3.  **环境说明**：后端服务已在 `8000` 端口启动，代码已 Push 至我的开发分支，`init.sql` 已同步更新。
